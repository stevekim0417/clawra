# CP-001: WhatsApp 한국 남친 봇 서비스 검토서

**문서번호:** CP-001
**작성일:** 2026-02-17
**상태:** 검토 완료 (비판적 리뷰 반영)
**프로젝트:** Clawra - WhatsApp Korean Boyfriend Bot

---

## 1. 서비스 개요

### 1.1 컨셉

영화 "Her"에서 영감을 받은 AI 남자친구 서비스.
구독자에게 WhatsApp을 통해 한국 남자친구 컨셉의 1:1 대화, 셀카, 영상을 제공한다.

### 1.2 핵심 가치

- 감성적이고 자연스러운 한국 남친 스타일 대화
- fal.ai 기반 얼굴 일관성 있는 셀카/영상 제공
- 유저가 원할 때 언제든 대화 가능 + 봇이 먼저 연락하는 자연스러운 관계감

### 1.3 서비스 흐름

| 구분 | 빈도 | 설명 |
|------|------|------|
| 봇 선톡 (Proactive) | 주 1~2회 | 아침 인사, 퇴근 안부 등 Template Message로 발송 |
| 유저 먼저 (Reactive) | 무제한 | 유저가 메시지 보내면 24시간 대화창 활성화, AI 자유 대화 |

> ### REVIEW COMMENT — 섹션 1
>
> **컨셉 자체는 시장 검증됨. 단, 포지셔닝 재고 필요.**
>
> - 한국의 Zeta(제타, Scatter Lab)가 이 모델을 이미 증명했다. **MAU 304만, 일평균 사용시간 2시간 46분**, 2025년 Q2 매출 52억원에 **영업이익률 17%**로 흑자 전환. "권서혁" 캐릭터 하나에 5,630만 건 상호작용. 컨셉이 먹힌다는 것은 확실하다.
> - 그러나 Zeta는 **한국 내수 + 자체 앱** 모델이다. 본 서비스는 **해외 시장 + WhatsApp 네이티브**라는 차별점이 있는데, 문서에 이 차별점이 명확히 서술되어 있지 않다. "영화 Her에서 영감"보다는 **"WhatsApp 네이티브 × K-boyfriend × 중남미"라는 3축 포지셔닝**을 전면에 내세워야 한다.
> - 선톡 주 1~2회는 적절하다. 업계 데이터상 선톡 빈도가 높으면 차단율이 급등한다.

## 2. 현재 코드베이스 분석

### 2.1 이미 보유한 인프라

| 기능 | 구현 상태 | 위치 |
|------|----------|------|
| WhatsApp 메시지 발송 | 구현됨 | OpenClaw Gateway (`localhost:18789`) |
| fal.ai 이미지 생성 | 구현됨 | `scripts/clawra-selfie.ts` |
| 얼굴 일관성 (Reference Image Edit) | 구현됨 | xAI Grok Imagine `/edit` 엔드포인트 |
| 페르소나 시스템 | 구현됨 | `SOUL.md` + `IDENTITY.md` 구조 |
| 멀티 플랫폼 메시징 | 구현됨 | Discord, Telegram, WhatsApp, Slack, Signal, Teams |

### 2.2 기술 스택

- **런타임:** Node.js >= 18.0.0 / TypeScript
- **이미지 AI:** fal.ai (`@fal-ai/client ^1.2.0`)
- **메시징 게이트웨이:** OpenClaw (WhatsApp JID 포맷: `번호@s.whatsapp.net`)
- **데이터베이스:** 없음 (현재 stateless 구조)

### 2.3 추가 개발 필요 사항

| 항목 | 현재 | 목표 | 우선순위 |
|------|------|------|---------|
| 유저 DB | 없음 | 구독자/대화기록/설정 관리 | P0 |
| 한국 남친 페르소나 | Clawra (여성) | 한국 남자 캐릭터 | P0 |
| 대화 엔진 | 없음 | Claude API + 컨텍스트 관리 | P0 |
| 스케줄러 | 없음 | 주 1~2회 선톡 Cron | P1 |
| 동영상 생성 | 없음 | fal.ai 영상 모델 연동 | P1 |
| 결제 시스템 | 없음 | Stripe 등 구독 결제 | P2 |

> ### REVIEW COMMENT — 섹션 2
>
> **기존 인프라 활용도 평가는 공감하나, "구현됨"의 수준을 과대평가하고 있을 가능성.**
>
> - WhatsApp 메시지 발송이 "구현됨"이라 했지만, 현재는 **단순 발송 기능**이지 대화형 Webhook 수신/처리/상태 관리가 된 것이 아니다. 남친봇은 양방향 실시간 대화가 핵심이므로 실질적으로 대화 인프라는 **새로 구축**에 가깝다.
> - fal.ai 이미지 생성도 `clawra-selfie.ts`는 **Clawra 여성 캐릭터 전용**으로 튜닝되어 있을 것이다. 한국 남성 캐릭터용 프롬프트/파이프라인은 처음부터 설계해야 한다.
> - **P0에 "대화 기록/컨텍스트 관리"가 빠져 있다.** 업계 1위 불만사항이 "기억력 없음"이다. Nomi.ai가 3-layer 메모리 시스템으로 차별화하고 있고, Zeta는 "데이터 플라이휠"로 대화 품질을 높이고 있다. 단순 DB 저장이 아니라 **메모리 아키텍처 설계가 P0**여야 한다.
> - 결제 시스템이 P2인 것은 동의. MVP에서는 수동 관리도 가능하다.

## 3. WhatsApp Business API 정책 검토

### 3.1 메시지 유형별 정책

| 유형 | 조건 | 비용 |
|------|------|------|
| **유저 먼저 (User-Initiated)** | 24시간 내 자유 응답 | 무료 (Service Conversation) |
| **봇 먼저 (Business-Initiated)** | Meta 승인 Template 필요 | 유료 (건당 $0.03~0.08) |

### 3.2 발송 한도 (Tier 시스템)

| Tier | 24시간 내 발송 가능 유저 수 |
|------|---------------------------|
| Tier 1 (신규) | 1,000명 |
| Tier 2 | 10,000명 |
| Tier 3 | 100,000명 |
| Tier 4 | 무제한 |

- 품질 점수 양호 시 자동 Tier 업그레이드
- 차단/신고 비율 높으면 Tier 다운그레이드 또는 번호 정지

### 3.3 정책 리스크

| 리스크 | 대응 방안 |
|--------|----------|
| 성인 콘텐츠 금지 | 셀카/대화 수위 제한, 콘텐츠 필터링 적용 |
| 스팸 판정 | 유저 opt-in 필수, 선톡 빈도 주 1~2회로 제한 |
| 비즈니스 인증 필요 | Meta Business 계정 사전 승인 |
| 번호 정지 위험 | 품질 모니터링, 차단율 추적 |

> ### REVIEW COMMENT — 섹션 3
>
> **이 섹션이 문서에서 가장 위험하게 낙관적인 부분이다. 심각한 플랫폼 리스크를 과소평가하고 있다.**
>
> #### Microsoft Copilot 퇴출 사례 (2026.01.15)
> - **Microsoft Copilot이 2026년 1월 15일자로 WhatsApp에서 퇴출되었다.** WhatsApp 정책 변경이 원인. Microsoft 규모의 기업도 WhatsApp 플랫폼에서 밀려났다는 것은 제3자 AI 봇의 WhatsApp 존속이 **Meta의 재량에 완전히 종속**된다는 의미다.
> - Meta는 자체 AI (Meta AI)를 WhatsApp, Instagram, Messenger에 이미 통합했다. **자사 AI와 경쟁하는 제3자 AI 봇을 허용할 이유가 없다.** 이것이 가장 큰 구조적 리스크다.
>
> #### "AI 남자친구" 서비스의 Meta 정책 위반 가능성
> - WhatsApp Commerce Policy는 **"dating services"와 관련된 비즈니스를 제한**할 수 있다. AI 남친봇이 이 범주에 해당하는지 명확하지 않으나, Meta가 해석의 재량을 가진다.
> - Template Message에 셀카 이미지를 포함하려면 **Media Message Template 승인**이 필요한데, "남자친구가 보내는 셀카" 컨셉으로 Meta 승인을 받을 수 있는지 불확실하다.
>
> #### 대응 방안 보완 필요
> - **Telegram을 백업 채널로 반드시 준비해야 한다.** Soulmaite.io가 Telegram 기반으로 유사 서비스를 운영 중이며, Telegram의 Bot API는 WhatsApp보다 훨씬 개방적이다.
> - **OpenClaw Gateway가 비공식(unofficial) WhatsApp 연동이라면 리스크가 더 크다.** 공식 WhatsApp Business API vs 비공식 연동인지 확인이 필요하다.

## 4. 시스템 아키텍처

### 4.1 전체 구조

```
┌─────────────────────────────────────────────────┐
│               WhatsApp Business API              │
│           (Meta Cloud API / OpenClaw)             │
└──────────┬──────────────────────┬───────────────┘
           │ Webhook (유저 메시지)  │ REST API (봇 발송)
           ▼                      ▲
┌──────────────────────────────────────────────────┐
│                메인 서버 (Node.js)                 │
│                                                   │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │
│  │  대화 엔진    │ │  스케줄러     │ │  미디어 생성  │ │
│  │ (Claude API) │ │  (Cron)     │ │  (fal.ai)   │ │
│  └──────┬──────┘ └──────┬──────┘ └──────┬──────┘ │
│         └───────────┬───┴───────────────┘         │
│                     ▼                             │
│  ┌─────────────────────────────────────────────┐  │
│  │                 Database                     │  │
│  │  users: 구독 상태, 폰번호, 설정              │  │
│  │  conversations: 대화 기록, 컨텍스트           │  │
│  │  media_cache: 생성된 사진/영상 URL            │  │
│  │  schedules: 선톡 스케줄, 발송 이력            │  │
│  └─────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

### 4.2 대화 엔진

```
유저 메시지 수신
    │
    ▼
유저 DB 조회 (구독 상태 확인)
    │
    ▼
최근 대화 기록 로드 (컨텍스트)
    │
    ▼
Claude API 호출
  - System: 한국 남친 페르소나
  - Context: 최근 대화 기록
  - User: 유저 메시지
    │
    ▼
응답에 셀카 요청 포함?
  ├─ Yes → fal.ai 이미지 생성 → 미디어 + 텍스트 발송
  └─ No  → 텍스트만 발송
    │
    ▼
대화 기록 저장
```

### 4.3 스케줄러 (선톡 시스템)

```
Cron Job (매일 실행)
    │
    ▼
오늘 선톡 대상 유저 조회
  - 구독 활성 상태
  - 마지막 선톡 후 3~4일 경과
  - 유저 타임존 기준 적절한 시간대
    │
    ▼
상황별 프롬프트 선택 (아침/저녁/주말 등)
    │
    ▼
fal.ai 셀카 생성 (상황에 맞는 프롬프트)
    │
    ▼
WhatsApp Template Message 발송
    │
    ▼
발송 이력 기록
```

> ### REVIEW COMMENT — 섹션 4
>
> **아키텍처 방향은 합리적이나, 핵심 컴포넌트가 누락되어 있다.**
>
> #### 누락된 아키텍처 요소
> 1. **메모리/컨텍스트 관리 레이어**: "최근 대화 기록 로드"만으로는 부족하다. 업계 선두 Nomi.ai는 3-layer 메모리 시스템(단기/중기/장기)을 사용한다. 최소한 다음이 필요:
>    - **단기 메모리**: 현재 세션 대화 (최근 20~50턴)
>    - **장기 메모리**: 유저 프로필, 취향, 중요 사건 요약 (벡터 DB 또는 요약본)
>    - **감정 상태 추적**: 유저의 현재 감정 상태에 맞는 응답 톤 조절
>
> 2. **콘텐츠 안전 필터**: 대화 엔진과 발송 사이에 **콘텐츠 필터링 레이어**가 없다. CA SB 243 (2026.01 시행), NY AI Companion Law (2025.11 시행) 등 규제 요건상 위기 상황 감지(자해/자살 언급) 및 대응 프로토콜이 필수다.
>
> 3. **응답 지연 시뮬레이션**: 즉각 응답은 "봇 느낌"을 준다. 1~5초 타이핑 딜레이 + "읽는 중" 상태 표시가 몰입감에 중요하다.
>
> #### 잘한 점
> - 단일 서버 구조는 MVP에 적합하다. 초기부터 마이크로서비스로 가면 오버엔지니어링이다.
> - 대화 엔진 → 미디어 생성 → 발송의 흐름은 깔끔하다.

## 5. fal.ai 미디어 생성 전략

### 5.1 얼굴 일관성 유지

현재 코드베이스의 `xai/grok-imagine-image/edit` 엔드포인트를 활용한다.

```
[기본 남친 이미지] → Reference Image로 고정 등록
         │
         ▼
  edit 모드로 상황별 이미지 생성
  → 동일 얼굴, 다른 배경/의상/표정
```

- **기본 이미지:** 한국 남자 캐릭터 기본 셀카 1장 생성 후 고정
- **상황별 생성:** 모든 후속 이미지는 edit 모드로 얼굴 일관성 유지

### 5.2 상황별 프롬프트 설계

| 상황 | 프롬프트 (예시) |
|------|----------------|
| 아침 셀카 | `Korean handsome boyfriend, morning selfie, messy bed hair, cozy bedroom, natural light` |
| 카페 | `Korean handsome boyfriend, cafe selfie, holding americano, warm lighting, casual outfit` |
| 운동 후 | `Korean handsome boyfriend, gym mirror selfie, workout clothes, post-workout glow` |
| 야경 | `Korean handsome boyfriend, city night view selfie, rooftop, romantic mood lighting` |
| 요리 중 | `Korean handsome boyfriend, kitchen selfie, cooking, wearing apron, smiling` |
| 출근길 | `Korean handsome boyfriend, morning commute selfie, suit, subway/street background` |

### 5.3 동영상 생성

fal.ai 영상 모델을 활용하여 짧은 클립 생성:

| 모델 후보 | 용도 |
|-----------|------|
| `minimax/video-01` | 짧은 인사 영상 (3~5초) |
| `kling-ai/v1.5` | 고품질 영상 |

- 생성된 이미지를 첫 프레임으로 사용하여 얼굴 일관성 유지
- 간단한 동작 (손 흔들기, 미소, 윙크 등)

> ### REVIEW COMMENT — 섹션 5
>
> **가장 기술적 불확실성이 높은 섹션. "edit 모드로 얼굴 일관성 유지"는 이론적으로 맞지만 실전에서 검증이 필요.**
>
> #### 얼굴 일관성의 현실
> - Reference Image Edit 방식은 **포즈/각도가 크게 바뀌면 일관성이 깨진다.** "아침 셀카(누워서)" → "운동 셀카(전신 거울)" → "야경(반측면)"처럼 다양한 각도를 요구하면 같은 사람으로 인식하기 어려울 수 있다.
> - Candy.AI, DreamGF 등 경쟁사는 이 문제를 **고정 3D 모델 + 렌더링** 또는 **LoRA 파인튜닝**으로 해결한다. edit 모드만으로는 한계가 있을 수 있다.
> - **MVP에서는 5~10개의 사전 생성 이미지 세트를 준비하고, 실시간 생성은 보조적으로 사용하는 전략**이 리스크를 줄인다.
>
> #### 동영상 생성의 현실성
> - fal.ai 영상 생성은 **한 건에 $0.10~0.50**으로 문서의 $0.10 추정보다 높을 수 있다. 또한 생성에 **30초~2분** 소요되어 실시간 대화 중 "잠깐만~" 하고 보내기엔 대기 시간이 길다.
> - 영상은 MVP에서 제외하고 Phase 3에서 **미리 생성된 영상 라이브러리** 방식으로 접근하는 것이 현실적이다.
>
> #### 잘한 점
> - 상황별 프롬프트 설계는 체계적이다. 이 수준의 프롬프트 라이브러리를 갖추면 다양한 상황 커버가 가능하다.
> - 이미지를 첫 프레임으로 영상 생성하는 접근은 올바른 방향이다.

> ### REVIEW COMMENT — 섹션 6
>
> **대화 예시는 자연스럽고 잘 만들어졌다. 공감한다. 단, 몇 가지 중요한 시나리오가 빠져 있다.**
>
> #### 추가해야 할 시나리오
> 1. **위기 상황 대응**: "나 죽고 싶어", "살기 싫다" 같은 메시지가 올 때의 대응. CA SB 243, NY AI Companion Law 모두 위기 대응 프로토콜을 요구한다. Character.AI의 미성년자 자살 소송(2024.10, 2026.01 합의)이 업계 전체에 경종을 울렸다. **이것은 선택이 아니라 법적 의무다.**
>
> 2. **반복 대화 대응**: 유저가 매일 같은 주제로 대화할 때(예: "오빠 오늘도 힘들었어") 봇이 매번 같은 반응을 하면 "반복적/얕은 대화"라는 업계 1위 불만사항에 해당한다. 변주를 주는 전략이 필요하다.
>
> 3. **언어 전환**: 중남미 타겟이면 대화가 **스페인어/포르투갈어**로 이루어진다. 한국어 예시만 있으면 실제 서비스와 괴리가 크다. "오빠" 대신 "oppa"를 쓸 건지, 스페인어에 한국어를 섞을 건지 언어 전략이 필요하다.
>
> #### 잘한 점
> - 대화 톤이 자연스럽고 "봇 느낌"이 적다.
> - 셀카 요청 시나리오에서 "잠깐만~" → 이미지 생성 → "방금 찍었어" 흐름이 몰입감 있다.

> ### REVIEW COMMENT — 섹션 7
>
> **비용 추정이 상당히 과소평가되어 있다. 여러 항목에서 현실과 괴리가 있다.**
>
> #### Claude API 비용 — 심각하게 과소평가
> - 문서: "30회/월 × $0.01 = $300/1,000명"
> - **현실**: Claude Sonnet 4.5 기준 input $3/MTok, output $15/MTok. 남친봇 대화 1회에 시스템 프롬프트(페르소나) + 대화 기록 컨텍스트 + 유저 메시지를 보내면 **input 2,000~5,000 토큰, output 200~500 토큰**이 사용된다.
>   - 1회 비용: input ~$0.01 + output ~$0.005 = **~$0.015**
>   - 단, 대화가 길어지면 컨텍스트가 누적되어 **10턴째에는 input이 10,000~20,000 토큰**으로 증가. 회당 비용이 **$0.03~0.06**까지 올라간다.
>   - 활성 유저가 하루 20~30턴 대화한다면(Zeta 평균 2시간 46분) 월 비용은 유저당 **$5~15**가 될 수 있다.
> - **Haiku 모델 사용 시** $0.25/$1.25 MTok으로 크게 절감 가능하나 대화 품질이 떨어질 수 있다. **Sonnet과 Haiku를 혼합**하는 전략 권장.
>
> #### 사용량 가정 — 과소평가
> - 문서: "30회/월/유저" → **하루 1회**에 불과
> - 현실: Zeta 유저는 **일평균 2시간 46분**, Replika 유저는 **일평균 2.7시간** 사용. 활성 유저는 **하루 20~50턴**, 월 **600~1,500턴**을 주고받는다.
> - 물론 모든 유저가 파워유저는 아니다. 하지만 "30회/월"은 거의 사용하지 않는 유저 기준이다. **활성 유저 평균 300~500회/월**로 재산정해야 한다.
>
> #### 구독 가격 — 중남미 시장과 괴리
> - 문서: "$9.99 ~ $14.99/월"
> - **현실**: LATAM 시장 조사 결과 디지털 구독 적정 가격대는 **$3~8/월**이다. 68%가 "현재 구독료를 감당하기 어렵다"고 응답. 아르헨티나에서는 63%가 가격 인상으로 구독을 해지한 경험이 있다.
> - **$9.99는 미국 가격이다.** LATAM에서 이 가격은 Netflix 프리미엄과 비슷한 수준이며, AI 남친봇에 이 가격을 지불할 사람은 극소수다.
> - **권장**: 기본 $4.99/월 (LATAM), $9.99/월 (US/EU). Pix(브라질), Mercado Pago, OXXO(멕시코 현금) 등 현지 결제 수단 지원 필수.

## 10. 결론

### 기술적 타당성: **가능**

- 현재 Clawra 코드베이스에 WhatsApp 메시징과 fal.ai 이미지 생성 인프라가 이미 구현되어 있음
- 추가 개발 범위는 DB, 대화 엔진, 스케줄러, 결제 시스템
- 기존 코드 구조를 확장하여 MVP 2~3주 내 개발 가능

### 사업적 타당성: **유망**

> - "MVP 2~3주"는 **메모리 시스템과 안전 필터를 포함하면 4~6주**로 보는 것이 정확하다.
>
> #### 사업적 타당성: **"유망"에서 "조건부 유망"으로 하향 조정**
>

> - WhatsApp × K-boyfriend × LATAM 교차점은 **실제 블루오션**이다. 조사 결과 이 교차점을 직접 공략하는 경쟁자는 없다.

> - Zeta 사례가 비즈니스 모델 자체를 검증한다. 문제는 "될까?"가 아니라 "어떻게?"다.
>
> **부정 요인 (신규)**:
> - **플랫폼 리스크가 사업 존속을 위협한다.** WhatsApp에서 쫓겨나면 끝이다. 반드시 멀티플랫폼(Telegram 병행) 전략이 필요하다.
> - **마진율 85~90%는 환상이다.** 현실적 비용 추정 시 LATAM $4.99 가격대에서는 **적자 또는 박리다매**. Haiku 모델 + 캐싱 최적화가 선행되어야 수익 가능.

> - **리텐션이 사업의 성패를 가른다.** 업계 30일 리텐션 13~50%. 메모리 시스템 없으면 유저는 2주 내 떠난다.
